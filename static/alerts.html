How do I find my validator public key?

Your validator public key starts with nH and uniquely identifies your validator on the network. Here's how to find it:

  1. Check your validator's configuration file — the key is usually in validators.txt or was shown when you first set up your node.
  2. Look at the leaderboard — find your validator by domain name and copy the public key.
  3. Or run this on your server:
    on your validator server
    # Get your validator's public key from the scoring API curl -s https://dashboard.pftoligarchy.com/api/scores | python3 -c " import sys,json for v in json.load(sys.stdin)['validators']: if v.get('domain') and 'YOUR_DOMAIN' in v['domain']: print(v['public_key'])"

    Replace YOUR_DOMAIN with your validator's domain name.

How do I find my node key?

Your node key starts with n9 and identifies your node on the peer network. SSH into your server and run:

on your validator server
docker exec postfiatd curl -s -X POST http://127.0.0.1:5005 \ -H 'Content-Type: application/json' \ -d '{"method":"server_info","params":[{}]}' \ | python3 -c "import sys,json; print(json.load(sys.stdin)['result']['info']['pubkey_node'])"

Paste the output (starts with n9) into the field above.

Note: For full security, we recommend also running the verification command below after subscribing. This proves you control the node and prevents anyone else from claiming your key.

How do I create a Discord webhook?

A webhook lets us send messages to your Discord channel. It takes about 30 seconds to set up:

  1. Open Discord and right-click your server name at the top of the channel list. Select Server Settings.
  2. In the left sidebar under Apps, click Integrations.
  3. Click Webhooks, then click New Webhook.
  4. Give it a name (e.g. "Validator Alerts"), and use the Channel dropdown to pick which channel receives alerts.
  5. Click Copy Webhook URL and paste it into the field above.

Webhook URLs look like: https://discord.com/api/webhooks/123.../abc...
Note: Do NOT use the gear icon at the bottom-left of Discord — that opens User Settings, not Server Settings.

How to verify your node key

To prevent anyone from claiming someone else's node, we verify you actually control it. You run a command from your validator server — because only you have SSH access, we can check that the request comes from your node's IP address. This proves ownership.

  1. SSH into your validator server
    your local terminal
    ssh root@YOUR_SERVER_IP

    Replace YOUR_SERVER_IP with your validator's IP address.

  2. Get your node key — this is your node's network identity (starts with n9)
    on your validator server
    # Get your node's public key docker exec postfiatd curl -s -X POST http://127.0.0.1:5005 \ -H 'Content-Type: application/json' \ -d '{"method":"server_info","params":[{}]}' \ | python3 -c "import sys,json; print(json.load(sys.stdin)['result']['info']['pubkey_node'])"

    Copy the output — you'll need it for the next step.

  3. Run the verification command — replace the two placeholder values with your actual keys
    on your validator server
    # Verify ownership — must be run from your validator server curl -X POST https://dashboard.pftoligarchy.com/api/alerts/verify-node \ -H 'Content-Type: application/json' \ -d '{"validator_key":"YOUR_VALIDATOR_KEY","node_key":"YOUR_NODE_KEY"}'

    What this does: Sends your two public keys to our API. We check that the request comes from the same IP address where your node lives on the network. Since only you have SSH access to your server, this proves you control the node.

    What it does NOT do: It doesn't send passwords, private keys, or any secrets. Both keys are public identifiers already visible to the network — this just verifies you're the one linking them.

  4. Look for the success message
    expected response
    {"message":"Verified! Your node key has been linked to your validator..."}

    If you get an IP mismatch error, make sure you're running the command on your validator server, not your laptop.

Is this safe? Both your validator key and node key are public identifiers — they're visible to the entire network already. This command doesn't expose any secrets. The only thing being verified is that you have SSH access to the machine running that node.
